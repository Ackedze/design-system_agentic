# Agentic Files

Этот документ описывает файлы, которые Apollo и агент используют как дополнительный контекст дизайн-системы: контракты компонентов, правила аудита, агентский контекст, примеры и индексы.

## Applicability

Каждый JSON-файл, используемый Apollo или агентом как каталог, индекс или контракт, должен содержать top-level блок `applicability`.

```json
{
  "applicability": {
    "channels": ["b2b"],
    "platforms": ["desktop", "mobile-web"],
    "source": "path-policy",
    "updatedAt": "2026-07-04"
  }
}
```

### Поля

| Поле | Тип | Назначение |
| --- | --- | --- |
| `channels` | `string[]` | Каналы, где файл применим. Сейчас используются `b2b` и `universal`. |
| `platforms` | `string[]` | Платформы, где файл применим: `desktop`, `mobile-web`, `ios`, `android`. |
| `source` | `string` | Источник политики маркировки. Для текущей миграции используется `path-policy`. |
| `updatedAt` | `string` | Дата последнего массового обновления применимости в формате `YYYY-MM-DD`. |

### Правила маркировки

| Область файлов | `channels` | `platforms` |
| --- | --- | --- |
| `JSONS/web/components/web-corp/**` | `b2b` | `desktop`, `mobile-web` |
| `JSONS/web/components/web-core/**` | `universal` | `desktop`, `mobile-web` |
| `desing-system_abm/JSONS/abm/ios/**` | `universal` | `ios` |
| `desing-system_abm/JSONS/indexes/abm/ios/**` | `universal` | `ios` |
| `desing-system_abm/JSONS/abm/android/**` | `universal` | `android` |
| `desing-system_abm/JSONS/indexes/abm/android/**` | `universal` | `android` |
| `design-system_ab/JSONS/images/**` | `b2b` | `desktop`, `mobile-web` |
| `design-system_ab/JSONS/indexes/images/**` | `b2b` | `desktop`, `mobile-web` |
| `design-system_ab/JSONS/icons/**` | `universal` | `desktop`, `mobile-web`, `ios`, `android` |
| `design-system_ab/JSONS/indexes/icons/**` | `universal` | `desktop`, `mobile-web`, `ios`, `android` |
| `Spacing -- Spacing component.json` и его index-файл | `universal` | `desktop`, `mobile-web` |

`web-core` помечается как `universal`, потому что это базовые web-компоненты и они не должны быть связаны только с B2B-каналом. Если компонент из `web-core` получает более узкую область применения, это нужно фиксировать точечно в его package-документах.

Markdown-файлы внутри component package не получают отдельный frontmatter с применимостью. Они наследуют область применения от соседних JSON-файлов того же package. Это оставляет JSON единственным машинно-читаемым источником правды.

## Типы Файлов

В заголовках ниже `✅` означает, что файл уже используется текущей сборкой Apollo runtime. Такие файлы выделены жирным.

### **`componentContractIndex.json` ✅**

Глобальный индекс агентских пакетов компонентов.

Где применяется: `figma`. Это runtime-входная точка для Figma-плагина Apollo. Агент не должен использовать индекс как нормативный источник.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `baseUrl` | Базовый URL, от которого Apollo докачивает package-файлы. |
| `packages` | Список компонентных пакетов, их путей и Figma-ключей. |
| `components` | Быстрый lookup по component key. |
| `aliases` | Альтернативные имена и ключи для поиска пакета. |

Целевое использование:

- Apollo загружает индекс один раз при старте или перед проверкой.
- По ключам компонентов из выбранной области Apollo определяет, какие package-файлы нужно докачать.
- Агент не должен использовать индекс как нормативное правило. Индекс только маршрутизирует к нужным файлам.

### **Raw catalog JSON ✅**

Полный или нормализованный каталог Figma-компонентов. Примеры: `Web _ Core -- Button.json`, `Icons -- corp.json`, `Spacing -- Spacing component.json`, `iOS _ Basic -- Spacing.json`.

Где применяется: `figma`. Текущий Apollo runtime использует raw-каталоги как основной baseline для проверки выбранной области. В `agentic` raw-каталог не должен передаваться агенту целиком; для агента нужны сжатые производные документы: `rules.json`, `agent-context.json`, `composition-contract.json` или slices из них.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `kind` | Тип каталога, обычно `catalog`. |
| `source` | Метаданные библиотеки, файла, времени генерации и состава каталога. |
| `components` | Компоненты, варианты, ключи, структура, properties и baseline-значения. |
| `elements` | Общие элементы/слои, если они вынесены отдельно от компонентов. |

Целевое использование:

- Apollo загружает raw-каталоги по reference/index-файлам и использует их для deterministic diff.
- Athena CLI должна обновлять raw-каталоги атомарно вместе с индексами и связанными agentic package-файлами.
- Агент не должен выводить нормативные правила напрямую из raw-каталога без `rules.json` или паттерна.

### **Catalog index JSON ✅**

Индекс конкретного raw-каталога. Обычно имеет суффикс `.index.json`.

Где применяется: `figma`. Текущий Apollo runtime использует index-файлы для быстрого поиска нужных raw-каталогов и component keys. В `agentic` эти индексы не являются нормативным контекстом.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `kind` | Тип индекса, обычно `component-index`. |
| `catalogName` | Имя каталога. |
| `fileName` | Имя raw JSON-файла, к которому относится индекс. |
| `library` | Название библиотеки. |
| `pageName` | Название страницы/набора в Figma. |
| `source` | Метаданные источника. |
| `components` | Сжатый список компонентов, ключей, имён, статусов и платформенных признаков. |

Целевое использование:

- Apollo использует index-файлы для lazy-loading raw-каталогов по выбранной области.
- Индекс должен обновляться при каждом изменении соответствующего raw-каталога.
- Агент не должен использовать index-файл как источник рекомендаций.

### `contract.generated.json`

Сгенерированный компактный контракт компонента на основе raw-каталога Figma.

Где применяется: `figma`. Целевое место файла — runtime-слой Apollo, где он должен помогать строить effective baseline компонента. В текущей сборке файл ещё не является активным source of truth. В `agentic` его можно использовать только как исходник для генерации более компактного `agent-context.json`, но не передавать агенту целиком.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `source` | Исходный каталог, библиотека и время генерации. |
| `compactBaselineLegend` | Справочник сокращений и принципов сжатия baseline. |
| `contractScope` | Границы контракта: компонент, библиотека, тип пакета. |
| `anatomySignature` | Стабильная структура компонента и вложенных узлов. |
| `components` | Сжатые эталонные состояния, варианты, слои и параметры. |

Целевое использование:

- Целевая роль Apollo — использовать файл как дополнительный эталон для сложных и составных компонентов.
- Этот файл не редактируется вручную.
- Если raw-каталог изменился, файл должен быть пересобран пайплайном Athena CLI.

### `contract.overrides.json`

Ручной слой поверх сгенерированного контракта. Нужен для случаев, где машинная структура недостаточно объясняет намерение компонента.

Где применяется: `figma`, `agentic`. В `figma` файл должен уточнять модель компонента до diff/classification: public API, slots, reset model и semantic ownership. В `agentic` он может использоваться как объяснение допустимой настройки компонента. В текущей сборке Apollo файл ещё не читается напрямую.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `appliesTo` | Какой компонент или пакет уточняется. |
| `publicApi` | Разрешённые публичные параметры компонента. |
| `anatomySemantics` | Семантика слотов и вложенных DS-инстансов. |
| `resetModel` | Как разделять сброс параметров компонента и параметров слоя. |

Целевое использование:

- Целевая роль Apollo — использовать файл при интерпретации diff и при построении групп сброса.
- Целевая роль агента — использовать его как объяснение допустимой настройки компонента.
- Этот файл не должен дублировать `rules.json`: overrides описывает модель компонента, а не список нарушений.

### **`composition-contract.json` ✅**

Контракт композиции составного компонента.

Где применяется: `figma`. Текущий Apollo runtime уже загружает файл через `componentContractIndex.json` и использует его для contract-aware diff/rebase у составных компонентов. В `agentic` файл может быть вспомогательным источником для генерации пояснений, но не должен заменять `agent-context.json`.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `composition` | Какие nested DS-инстансы входят в компонент. |
| `slotMap` | Карта слотов и ожидаемых вложенных областей. |
| `baselineOwnership` | Кто владеет baseline: host-компонент или вложенный компонент. |
| `allowedRemaps` | Разрешённые переназначения параметров внутри композиции. |

Целевое использование:

- Apollo использует файл, чтобы не считать корректные настройки host-компонента кастомизациями вложенных компонентов.
- Особенно важен для компонентов-обёрток: `TitleView`, `TabsView`, `ButtonGroup`, таблицы, header-компоненты.
- Если файла нет, Apollo должен работать по raw-каталогу и `contract.generated.json`, но результат может быть менее точным для составных компонентов.

### **`rules.json` ✅**

Детерминированные правила компонента.

Где применяется: `figma`, `agentic`. В `figma` текущий Apollo runtime уже загружает файл, матчится с diff и добавляет найденные правила в `*_agent.json`. В `agentic` эти правила используются агентом как компонентный нормативный контекст через `componentRules` внутри агентского отчёта.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `rules` | Список правил с `ruleId`, `severity`, `source`, `appliesTo`, `checkType`, `ruleText`. |

Целевое использование:

- Apollo добавляет релевантные правила в `*_agent.json`, если найденная кастомизация относится к этому компоненту или свойству.
- Агент использует `rules.json` как компонентный нормативный контекст.
- `severity: error` может повышать критичность, если правило точно совпадает с отклонением.
- `severity: warning` требует рекомендации, но не всегда означает запрет.
- `severity: info` не является нарушением, но должно объясняться агентом, если правило относится к найденному факту.

Правило не должно быть примером. Если правило описывает только техническую природу diff, формулировка должна явно говорить, что это классификация, а не запрет.

### `audit-mapping.json`

Файл настройки того, как Apollo должен классифицировать и показывать результаты аудита для компонента.

Где применяется: `figma`, `agentic`. В `figma` файл должен управлять классификацией, группировкой, порядком и reset-action для diff. В `agentic` может использоваться как технический контекст о том, почему Apollo сгруппировал находку именно так. В текущей сборке Apollo файл ещё не читается напрямую.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `classification` | Как относить diff к категориям Apollo. |
| `groupingOrder` | Порядок группировки находок в UI. |
| `evidencePolicy` | Какие факты считать достаточным доказательством. |
| `baselinePolicy` | Как выбирать expected-значение при variant/property override. |

Целевое использование:

- Целевая роль Apollo — использовать файл на этапе построения audit view и `*_agent.json`.
- Целевая роль агента — использовать его как технический контекст, но не как нормативный паттерн.
- Если Apollo уже корректно определяет параметр кастомизации, `audit-mapping.json` всё равно полезен для порядка, группировки и evidence.

### `agent-context.json`

Человекочитаемый контекст компонента для агента.

Где применяется: `agentic`. Это основной компактный контекст для ответа агента: назначение компонента, anatomy, links, interpretation rules и ссылки на `rules.json`. Сейчас файл опубликован в `componentContractIndex.json` как artifact, но текущий Apollo runtime его ещё не загружает и не добавляет в `*_agent.json`.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `summary` | Краткое назначение компонента. |
| `publicComponentProperties` | Публичные свойства и ожидаемый способ настройки. |
| `anatomy` | Слоты, вложенные компоненты и важные слои. |
| `auditInterpretation` | Как агент должен читать находки Apollo по этому компоненту. |
| `patternLinks` | Связанные паттерны и Figma-ссылки. |
| `exampleInterpretations` | Примеры корректной интерпретации diff. |
| `ruleReferences` | Ссылки на правила из `rules.json`. |

Целевое использование:

- Агент использует файл для объяснения дизайнеру, почему Apollo нашёл отклонение и как его исправлять.
- Файл не должен придумывать запреты, которых нет в `rules.json` или паттернах.
- Если `agent-context.json` содержит примеры, агент обязан отличать пример от правила.

### `examples.json`

Набор тестовых и обучающих примеров для компонента.

Где применяется: `agentic`. Файл нужен для тестирования и настройки качества агентской интерпретации. В `figma` runtime он не нужен и не должен докачиваться при обычной проверке.

Основные разделы:

| Раздел | Назначение |
| --- | --- |
| `examples` | Сценарии изменения компонента, ожидаемый diff и желаемая интерпретация. |

Целевое использование:

- Используется для ручного и автоматического тестирования качества интерпретации.
- Может помогать агенту с формулировками, но не является источником новых правил.
- Если пример конфликтует с `rules.json` или паттерном, приоритет у правила или паттерна.

### Component `README.md`

Краткая документация конкретного пакета компонента.

Где применяется: `agentic`. Это authoring-документация для людей, которые поддерживают package. В `figma` runtime не используется.

Целевое использование:

- Для разработчиков Apollo, Athena CLI и авторов контрактов.
- Может описывать ручные решения, известные ограничения, источники и принятые допущения.
- Не должен быть единственным источником машинно-читаемых правил.

## Приоритет Источников

Если источники конфликтуют, используется такой порядок:

1. `rules.json` с `severity: error` и точным совпадением с фактом Apollo.
2. Паттерн дизайн-системы с точным правилом.
3. `contract.overrides.json` и `composition-contract.json`.
4. `contract.generated.json`.
5. `audit-mapping.json`.
6. `agent-context.json`.
7. `examples.json`.
8. Component `README.md`.

Примеры, summary и пояснения не могут повышать критичность сами по себе. Повышение критичности допустимо только по точному правилу или явному компонентному `severity`.

## Правила Поддержки

- Все JSON-файлы должны быть валидными и форматироваться с отступом 2 пробела.
- `componentKey` должен оставаться стабильным между файлами одного component package.
- `applicability` обновляется при переносе файла между репозиториями, каналами или платформами.
- Generated-файлы пересобираются Athena CLI, curated-файлы редактируются вручную.
- Raw-каталоги не должны дублироваться внутри component package, если они уже хранятся в основном каталожном репозитории.
- Apollo должен докачивать только файлы, подходящие под выбранный канал и платформу.
